# 测试规范（模板）

> 本文件为通用测试规范模板，需按项目技术栈补充具体工具和配置。

---

## 测试覆盖率要求

- **最低覆盖率**：语句覆盖 80%+
- **核心模块**：工具函数、Hooks/Composables、Stores 必须测试

---

## 测试类型

| 类型 | 范围 | 工具 |
|------|------|------|
| 单元测试 | 独立函数、工具类 | （按项目填写） |
| 组件测试 | UI 组件渲染、交互 | （按项目填写） |
| 集成测试 | API 接口、数据流 | （按项目填写） |

---

## 测试文件规范

- 文件命名：`{module}.spec.ts`，与被测试文件同目录
- 所有 `describe` 和 `it` 描述必须使用**中文**
- 命名格式：`应该{预期结果}` 或 `当{条件}时应该{结果}`

```typescript
import { describe, expect, it } from 'vitest'
import { myFunction } from './myModule'

describe('myModule - 功能描述', () => {
  it('应该在正常输入时返回正确结果', () => {
    expect(myFunction('test')).toBe('expected')
  })

  it('应该在异常输入时抛出错误', () => {
    expect(() => myFunction(null)).toThrow()
  })
})
```

## Mock 原则

- 使用工厂函数创建测试数据，避免共享可变状态
- mock 外部依赖（API、浏览器 API）
- 集成测试优先 mock 网络响应，不直接 mock 内部模块
- 不要在同一条测试链路里混用不同 mock 策略，避免失去测试意义

```typescript
// 数据工厂函数
function createMockItem(overrides?: Partial<Item>) {
  return {
    id: 'item_001',
    name: '默认名称',
    ...overrides,
  }
}
```

---

## 禁止事项

- ❌ 禁止使用英文注释和英文测试描述
- ❌ 禁止一个 `it` 测试多个行为
- ❌ 禁止测试实现细节（内部变量）
- ❌ 禁止过度 mock（失去测试价值）
- ❌ 禁止在单元测试中包含真实网络请求

## TDD 反模式（来自 Superpowers 的 5 条铁律）

TDD 不只是"先写测试再写代码"，还要求测试验证的是**真实行为**而非 mock 行为。

### 反模式 1：测试 Mock 行为

```typescript
// ❌ 测试 mock 的存在而非组件真实行为
test('渲染侧边栏', () => {
  render(<Page />);
  expect(screen.getByTestId('sidebar-mock')).toBeInTheDocument();
});

// ✅ 测试真实组件或不使用 mock
test('渲染侧边栏', () => {
  render(<Page />);
  expect(screen.getByRole('navigation')).toBeInTheDocument();
});
```

**门禁**：断言 mock 元素前先问——我是在测试真实组件行为还是 mock 的存在？

### 反模式 2：仅为测试添加的方法

```typescript
// ❌ destroy() 仅在测试中使用，污染生产类
class Session {
  async destroy() { /* 仅测试使用 */ }
}

// ✅ 测试工具函数处理测试清理
export async function cleanupSession(session: Session) { /* ... */ }
```

**门禁**：向生产类添加方法前先问——这个方法仅测试使用吗？是 → 放到测试工具文件。

### 反模式 3：不理解依赖就 Mock

```typescript
// ❌ Mock 了测试所依赖的副作用
vi.mock('ToolCatalog', () => ({
  discoverAndCacheTools: vi.fn().mockResolvedValue(undefined) // 破坏了下游逻辑！
}));

// ✅ Mock 最底层的外部操作，保留中间层行为
vi.mock('MCPServerManager'); // 只 mock 慢的服务器启动部分
```

**门禁**：Mock 任何方法前先问——真实方法有哪些副作用？本测试依赖其中哪些？不确定时先用真实实现跑一遍。

### 反模式 4：不完整 Mock

```typescript
// ❌ 只 mock 了当前测试用到的字段
const mockResponse = { status: 'success', data: { userId: '123' } };
// 下游代码访问 response.metadata.requestId → 崩溃

// ✅ Mock 完整的数据结构
const mockResponse = {
  status: 'success',
  data: { userId: '123' },
  metadata: { requestId: 'req-789', timestamp: Date.now() },
};
```

**铁律**：Mock 必须是真实 API 返回的**完整**结构，不是只填测试当前用到的字段。不完整 Mock 会静默失败。

### 反模式 5：集成测试事后补

- ❌ 实现完成 → 无测试 → "可以测试了"
- ✅ TDD 循环：写失败测试 → 实现 → 重构 → 然后才说完成

**铁律**：测试是实现的一部分，不是可选的后续步骤。

### Mock 复杂度警告

以下信号表明 Mock 过度：Mock 设置比测试逻辑还长、Mock 了所有东西才能跑测试、Mock 缺少真实组件的方法、测试在移除 mock 时失败。

**考虑**：复杂 Mock 时，集成测试通常更简单有效。

## 验收标准规范（按子任务分组）

Planner 输出验收标准、Tester 核查时统一使用以下格式：

```markdown
#### 子任务 N：[名称]

| 场景分类 | 场景 | 前置条件 | 操作步骤 | 预期结果（可量化） | 不应发生 | 验证方式 |
|---------|------|---------|---------|-----------------|---------|---------|
```

- `场景分类` 只能填写：`正向 / 异常 / 边界 / 回归`
- `验证方式` 只能填写：`自动 / 手动 / N/A（原因）`
- `自动`：可直接转化为自动化用例
- `手动`：必须写出可复现的操作步骤和观察点
- 不适用时注明 `N/A（原因）`，不得省略

---

## 最小覆盖规则

- 每个子任务至少 1 条 `正向` 验收项，无豁免
- 命中接口调用、异步切换、状态写入、路由切换任一项时，至少再补 1 条 `异常` 或 `边界`
- 存在影响面地图消费方时，至少补 1 条 `回归`
- 纯样式微调、纯文案修改、纯静态展示可豁免 `异常 / 边界 / 回归`，但必须写 `N/A（原因）`
- 若子任务包含多个独立行为，不能只写 1 条总验收，需拆到行为级

---

## 场景覆盖要求

`正向`：必填，无豁免。

`异常`（纯展示类可豁免）：网络超时/500/404/断网、空列表/null 字段/格式错误、操作中途离开/token 过期/并发冲突、空值/非法字符/超长/特殊符号、无权限/角色不符。

`边界`（纯 UI 微调可豁免）：数据量空/单条/1000+/极长文本、快速连点/多操作并发、从 loading/error/empty 进入操作、请求未返回时切换/多请求并发。

`回归`（全新功能无调用方可豁免）：根据影响面地图消费方推导，每模块至少 1 个用例。

---

## 潜在问题检查清单（Planner 填写，Tester 核查）

格式：`存在 / 不存在 / N/A（原因）`，不得留空。

绑定规则：

- 标记为 `存在` 的项，必须在验收标准表中至少有 1 条对应验证项
- 标记为 `N/A` 的项，必须说明为何当前子任务不适用
- Tester 核查时如发现清单为 `存在` 但验收标准未覆盖，应直接打回 Planner 补齐

```text
□ 重复请求：并发点击/重复渲染触发多次请求？
□ 竞态条件：异步操作顺序不确定导致数据覆盖？
□ 内存泄漏：事件监听器/定时器/watch 在卸载时未清理？
□ 状态残留：页面卸载后 store 需要重置，重置逻辑是否完整？
□ 数据丢失：操作中途退出是否丢失未提交数据？
□ 外部依赖：是否处理了外部服务未就绪/调用失败的情况？
□ 三态处理：loading / error / empty 是否均有明确 UI 反馈？
□ 不可逆操作：删除/提交等是否有二次确认保护？
□ 副作用扩散：是否会意外触发其他模块的状态变化或请求？
```

---

## 验证方式要求

- **优先 Playwright**：凡是涉及浏览器渲染、用户交互、DOM 断言、E2E 流程的场景，优先使用 Playwright 进行自动化测试；只有 Playwright 无法覆盖的场景（如纯 Node.js 逻辑、CLI 工具）才回退到其他测试工具
- 优先自动化：稳定可重复的逻辑判断、接口行为、状态变更、边界输入，优先写成自动验证
- 必须手动：视觉样式、真机交互、外部联调、动画体感、复杂画布操作，可使用手动验证
- 不能自动化时，Planner 要写清手动原因；Tester 要按步骤逐项记录结果
- 同一条验收项不得只写"人工验证"或"测试通过"，必须能看出验证对象和判断标准

---

## Tester 验证质量门禁

功能看起来能跑只是最低要求，不等于测试通过。Tester 必须审查“验证是否可信”，不能只照着正向路径点一遍。

Tester 必须基于验收标准、影响面、改动文件、测试命令和实际执行结果判断：

- **验收完整性**：正向、异常、边界、回归是否按最小覆盖规则覆盖；缺失项是否有合理 `N/A（原因）`
- **影响面覆盖**：所有受影响模块、调用方、状态流、接口、UI 入口是否有对应验证
- **执行真实性**：必须明确实际运行的命令、手动步骤、环境和结果；不得只写“已验证 / 测试通过”
- **证据充分性**：关键断言、失败日志、截图、录屏、接口响应、DOM 状态或数据变化必须能支撑结论
- **回归风险**：改动周边的旧能力是否被验证；高风险回归未覆盖时不得放行
- **可复现性**：失败用例必须包含前置条件、操作步骤、期望结果、实际结果和复现稳定性
- **未测风险**：无法执行、无法覆盖或依赖外部环境的项必须显式列出并说明影响

判定规则：

| 严重级别 | 判定标准 | 处理 |
|----------|----------|------|
| CRITICAL | 核心验收失败、关键测试命令失败、数据安全/主流程不可用 | 阻断 |
| HIGH | 功能表面可用但关键异常、边界、回归或影响面未验证；无可复现证据 | 阻断 |
| MEDIUM | 局部覆盖不足或证据不完整，但不影响当前合入判断 | 记录并建议补充 |
| LOW | 测试描述、截图命名、报告格式等轻微问题 | 建议 |

若“主路径通过但关键风险未测”，Tester 必须至少给出 HIGH 或 MEDIUM 问题，不得以“功能可用”直接通过。HIGH 问题必须路由 developer 或 planner 补齐后复测。

Tester 输出必须包含：

```md
## Tester 结论

可通过：是 / 否
总体评价：[不仅描述功能是否可用，还要评价验证覆盖和证据质量]

### 覆盖矩阵
| 验收项 | 场景分类 | 验证方式 | 执行结果 | 证据 | 结论 |
|--------|----------|----------|----------|------|------|

### 阻断问题
- [CRITICAL/HIGH] 场景 — 问题、证据、复现步骤、建议修复方向

### 非阻断建议
- [MEDIUM/LOW] 场景 — 问题、影响、建议

### 未验证 / 无法验证
- [场景] — 原因、风险、建议后续验证方式

### 已执行命令 / 手动步骤
- [命令或步骤] — 结果

### 剩余风险
- [没有则写“无”]
```
