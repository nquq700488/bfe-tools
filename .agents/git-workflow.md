# Git 工作流规范

---

## 分支管理

| 分支 | 说明 |
|------|------|
| `main` | 生产分支，禁止直接推送 |
| `develop` | 开发分支，功能集成 |
| `feature/*` | 功能分支，从 develop 切出 |
| `fix/*` | 修复分支 |

（按项目实际分支策略调整）

---

## Commit Message 格式

遵循 [Conventional Commits](https://www.conventionalcommits.org/) 规范。

```text
<type>: <description>
```

| 类型 | 用途 |
|------|------|
| `feat` | 新功能 |
| `fix` | 修复 bug |
| `refactor` | 重构（无功能变化） |
| `docs` | 文档更新 |
| `test` | 测试相关 |
| `chore` | 构建/工具链 |
| `perf` | 性能优化 |
| `ci` | CI/CD 配置 |

- 描述使用中文，控制在 50 字以内
- 单次提交只包含一个逻辑变更

---

## 变更拆分规则

提交前必须先按文件和变更目的判断是否需要拆分。命中任一信号即应拆分为多次提交：

| 信号 | 说明 |
|------|------|
| 多个不相关目的 | 同时包含功能新增、bug 修复、样式调整、文档更新等不同目的 |
| 多个独立模块 | 变更跨越无直接依赖的模块，可独立交付 |
| 混合 type | 变更需要不同 commit type，如 `feat` + `fix` + `docs` |
| 可独立回滚 | 某部分变更有独立价值，应能单独 revert |

拆分执行规则：

- 每次提交只 stage 当前组文件，禁止 `git add .`
- 按依赖顺序提交：类型定义 / 底层逻辑 → 上层组件 → 测试 / 文档
- 每组提交都必须单独确认 type 和 description
- 不确定是否同一逻辑单元时，默认拆分或先问用户

---

## Pull Request 流程

1. 从最新 develop 切出分支
2. 完成功能并测试通过
3. 提交 PR 到 develop，填写变更内容和测试计划
4. Code Review 通过后合并

---

## 质量门禁

| 场景 | 处理 |
|------|------|
| `lint` 有 error | 不得提交，必须修复 |
| `lint` 有 warning | 必须修复，或在交付说明中明确说明为何可忽略 |
| 类型检查失败 | 不得提交，必须修复 |
| CI 失败 | 不得继续推进提交流程，需修复后重试 |

---

## 禁止事项

- ❌ 禁止直接推送 main/develop
- ❌ 禁止大而全的提交
- ❌ 禁止提交包含 `console.log` 的代码
- ❌ 禁止提交 mock 数据或临时测试数据（开发环境自用除外）
- ❌ 禁止 `--no-verify` 跳过 pre-commit hook
- ❌ 危险操作（`reset --hard`、`push --force`、`git rm`）必须先询问用户确认
