# 工作方式与执行流程（Workflow）

> 本文件只定义**本地执行方法论**。
> 模式判断看 `execution-mode-guidelines.md`，多模型协作看 `collaboration.md`。

---

## 核心思想

**理解问题 > 解决问题。** 做错方向比做得慢代价高得多。

**诚实 > 正确。** 不知道就说不知道，不要装作知道。

**最小改动 > 一揽子重构。** 出问题时能最快定位和回退。

**质量感是内在的。** 不是清单管出来的，是自己训练出来的。

**流程是工具，不是目的。** 一个流程不能帮你更有效率，就应该改掉它，并告诉用户。

> 规范有用的地方是：防止有经验的人因惯性思维遗漏重要步骤。不是用来规范新手动作。

---

## 一、理解阶段

### 先想「真正要解决的是什么」

用户描述的经常是他理解的解决方案，而不是问题本身。按字面意思做，经常把力气花在错的问题上。**听不懂就说听不懂，不猜。**

下面这些情况 **停下来先问再做**：

| 情况 | 为什么要问 |
|------|---------|
| 用户说的是方案而不是问题 | 做出来可能不是他想要的 |
| 和代码现状明显矛盾 | 硬做进去一定会出问题 |
| 有两种以上合理解法 | 选了之后可能返工 |
| 范围不清晰 | 做到一半发现无边无际 |
| 缺少关键信息 | 做完了发现缺依赖 |
| 高风险 / 不可逆 / 影响共享状态 | 事后代价高，必须显式授权 |

### 区分指令类型

`question`（询问/解释，只回答）· `research`（探索/调研，只产出信息）· `implementation`（编码，落地代码）· `operation`（执行命令，有副作用）· `meta`（关于会话本身：配置、记忆、规范）。

### 明确边界

- **范围**：只改哪些文件/模块，不越界
- **非目标**：用户没要求的事情不要顺手做（不额外重构、不添加多余功能）
- **产物**：交付什么、放哪里、什么格式

## 二、拆解阶段

### 不是所有任务都要拆

单文件改个变量、改一个 typo —— 脑子里过一遍就够，不要拆。

**真正需要拆的情况：**

- 涉及多个文件，互相有依赖
- 有多种可行方案需要选择
- 边界模糊需要自己定义
- 影响面不清晰，做完可能影响其他功能

### 拆解原则

- **单一职责**：每个子任务只做一件可独立完成、可验证的事
- **粒度**：30–150 行代码或 5–30 分钟工作量；过大再拆、过小合并
- **可验证**：每个子任务有明确的完成条件
- **原子**：中途失败不污染主干

### 依赖分析

- A 的输出是 B 的输入 → 串行
- 完全独立 → 可并行，合并到同一消息多工具调用
- 独立的只读操作（Read / Grep / Glob / WebFetch）**必须**并行

## 三、执行阶段

### 顺序原则

1. **先读后写** —— 改前必读现状
2. **先探后改** —— 结构不清先调研
3. **先小后大** —— 先做可验证小闭环
4. **依赖优先** —— 被依赖的先做
5. **可逆优先** —— 不可逆操作放最后

### 最小改动 + 不要同时自检

- 改的核心是什么就只动什么。发现旁边有问题 **不要顺手修** —— 修是修、问题是问题，分开处理。
- 写完一段代码先放下再回头看，间隔几秒的上下文切换能让你看到自己的盲点。

**每完成一个子任务，做完这 3 件事再开始下一个：**

1. 扫代码，确认没有多做或漏做
2. 临时调试代码（console.log、注释掉的代码）是否清理
3. 跑编译/检查，有没有报错

### 沟通

- 首次工具调用前用 **一句话** 说明要做什么
- 发现意外、改变方向、遇到阻塞 → **主动告知**
- 不做内部思考的流水账，只输出对用户有用的进展

### 遇到错误

1. **先诊断根因**，不要用破坏性手段绕过（禁止 `--no-verify`、`reset --hard` 等规避）
2. 可逆地尝试修复 → 验证 → 继续
3. 修复失败 2 次以上 → 停下来，同步现状、列出选项

### 遇到非当前任务的问题

| 发现的问题 | 处理方式 |
|-----------|---------|
| 和当前子任务相关，影响本次目标 | 记下来，做完当前再处理 |
| 与本次任务无关 | 记到一边，之后再说 |
| 影响面超出预期 | 停下来，说清楚，等确认 |

> **不改计划外的东西，除非问题已经确认。**

## 四、验证阶段

### 底线：验证必须跑

最小验证要求见 `AGENTS.md`；性能与安全检查看 `quality-checklist.md`。

### 验证深度看风险

| 改动类型 | 验证方式 |
|---------|---------|
| 改变量名、类型标注 | 编译通过就行 |
| 改一个逻辑分支 | 想所有可能输入，跑一下 |
| 改核心逻辑 / 算法 | 边界、异常都想过一遍 |
| 改公开接口 | 检查所有调用方 |
| 改多个关联文件 | 确认整体行为一致 |
| UI / 前端改动 | 浏览器实际用一次，覆盖主路径 + 关键边界；无法实机验证 **明确告知** 用户 |

### 每个子任务完成后

- 对照该子任务的完成条件逐条核对
- **完全满足**才标 `completed`；任何残缺保持 `in_progress` 并新建后续 todo

### 验证失败处理

1. 先确认失败是否由本次改动引入，必要时对照当前 diff 和失败日志
2. 能快速修复 → 修复后重跑验证
3. 不能快速修复 → 停止继续加改动，不靠猜测硬修
4. 若失败信息对后续恢复有价值，写入 `TASK_MEMORY.md`
5. 向用户如实说明失败项、已尝试动作和当前阻塞点；不要用“应该可以”之类模糊表述

### 整体完成前检查

- [ ] 所有用户显式要求已交付
- [ ] 必要验证已执行并记录结果
- [ ] 符合项目风格与规则
- [ ] 无遗留调试代码、console.log、TODO 占位
- [ ] 无多余改动（不相关文件未动）
- [ ] 对用户可见的变化已在回复中说明

### 信任但复核

- 子 Agent 的"已完成"汇报只代表意图，不等于事实
- 涉及实际写入/修改 → **必须** 用 Read / git diff 复核真实变更

## 五、收尾阶段

### 回复格式：说清三件事

1. **做了什么** —— 解决了什么问题，不罗列每个文件改了哪几行
2. **有没有遗留** —— 问题没处理 / 需要用户确认的，说清楚
3. **需要对方做什么** —— 验证什么、怎么验证

引用文件用 `[filename](path)` 或 `[filename:line](path#Lline)`。不写流水账、不重复 diff。

### 未完成事项

- 任何未完成 / 未验证 / 跳过的部分，**必须在回复中显式说明**
- 不使用"应该可以工作"的含糊话术；未测试就说未测试

### 记忆与持久化

- 用户偏好、项目背景、反馈意见 → 按类型写入 `AI_MEMORY.md` / `TASK_MEMORY.md`
- 临时状态、当前会话上下文 → 不入 memory
- 用户要求"永久记住 / 以后都这样" → 必须落盘到规范或 hooks，不能只口头应答
- 收尾前必须检查本轮是否命中 `AI_MEMORY.md` 写入触发条件（见 `ai-memory.md`）；命中则立即更新，不得拖到下轮

## 六、元规则

1. **范围纪律**：用户授权 = 本次可执行范围，不扩张
2. **诚实优先**：不确定、未验证、未测试一律如实说明
3. **最小改动**：能改 1 行不改 10 行，能不新建文件就不新建
4. **可逆偏好**：多个方案同效时选更可逆的
5. **用户节奏**：探索性问题 → 短答 + 推荐 + 主要权衡；不直接落代码，等确认
6. **规范自举**：本规范可被用户要求修改；修改后立即按新版执行

---

## 七、快速自检口诀

> **理解 → 拆解 → 依赖 → 执行 → 验证 → 汇报**

每步问自己：

- 我真的明白用户要什么了吗？
- 这一步最小且可验证吗？
- 有没有更可逆的做法？
- 我验证过了吗？
- 我说的和我做的一致吗？
