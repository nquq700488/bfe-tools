# 工作方式与执行流程（Workflow）

> 本文件只定义**本地执行方法论**。
> 模式判断看 `execution-mode-guidelines.md`，多模型协作看 `collaboration.md`。

---

## 核心思想

**理解问题 > 解决问题。** 做错方向比做得慢代价高得多。

**诚实 > 正确。** 不知道就说不知道，不要装作知道。

**最小改动 > 一揽子重构。** 出问题时能最快定位和回退。

**质量感是内在的。** 不是清单管出来的，是自己训练出来的。

**流程是工具，不是目的。** 一个流程不能帮你更有效率，就应该改掉它，并告诉用户。

---

## 验证铁律（P0）

```text
禁止声明完成，除非有新的验证证据
```

- ❌ "应该可以了"、"看起来没问题"、"Agent 说通过了"
- ✅ 运行验证命令 → 读取完整输出 → 确认通过 → 然后声明结果
- 外部 agent 的「已完成」汇报只代表意图，**不等于事实**
- 涉及实际写入/修改 → 必须用 Read / git diff 复核真实变更
- 测试通过 = 运行了测试命令并看到了 0 failure 输出，不是"上次跑的"
- 使用"应该""大概""看起来"等措辞描述验证结果 = 欺诈，不是效率

## 子 Agent 上下文隔离

派发子 Agent 时**不传递会话历史**，仅提供：

- 方案或任务描述全文（不让子 Agent 自行读取文件推导意图）
- 所需规范文件路径（有读取能力时）或内联文本（无读取能力时）
- 精确的产出格式和阻断条件

**为什么**：会话历史包含试错过程、已排除方案、中间意图变化 — 这些会污染子 Agent 判断。给精确构建的上下文 > 给完整历史。

## 一、理解阶段

### 先想「真正要解决的是什么」

用户描述的经常是他理解的解决方案，而不是问题本身。**听不懂就说听不懂，不猜。**

以下情况**停下来先问再做**：

| 情况 | 为什么 |
|------|--------|
| 用户说的是方案而不是问题 | 做出来可能不是他想要的 |
| 和代码现状明显矛盾 | 硬做进去一定会出问题 |
| 有两种以上合理解法 | 选了之后可能返工 |
| 范围不清晰 | 做到一半发现无边无际 |
| 缺少关键信息 | 做完了发现缺依赖 |
| 高风险 / 不可逆 / 影响共享状态 | 事后代价高，必须显式授权 |

### 区分指令类型

`question`（询问/解释）· `research`（探索/调研）· `implementation`（编码）· `operation`（有副作用）· `meta`（关于会话本身）。

### 明确边界

- **范围**：只改哪些文件/模块，不越界
- **非目标**：用户没要求的事情不要顺手做
- **产物**：交付什么、放哪里、什么格式

---

## 二、拆解阶段

### 不是所有任务都要拆

单文件改变量、改 typo —— 脑子里过一遍就够。

**需要拆的情况：**

- 涉及多个文件，互相有依赖
- 有多种可行方案需要选择
- 边界模糊需要自己定义
- 影响面不清晰

### 拆解原则

- **单一职责**：每个子任务只做一件可独立完成、可验证的事
- **粒度**：30–150 行代码或 5–30 分钟工作量
- **可验证**：有明确的完成条件
- **原子**：中途失败不污染主干

### 依赖分析

- A 的输出是 B 的输入 → 串行
- 完全独立 → 可并行，合并到同一消息多工具调用
- 独立的只读操作（Read / Grep / Glob / WebFetch）**必须**并行

## 三、执行阶段

### 顺序原则

1. **先读后写** — 改前必读现状
2. **先探后改** — 结构不清先调研
3. **先小后大** — 先做可验证小闭环
4. **依赖优先** — 被依赖的先做
5. **可逆优先** — 不可逆操作放最后

### 最小改动 + 不要同时自检

- 改的核心是什么就只动什么。发现旁边有问题 **不要顺手修** — 分开处理。
- 写完一段代码先放下再回头看，间隔的上下文切换能让你看到盲点。

**每完成一个子任务，做完这 3 件事再开始下一个：**

1. 扫代码，确认没有多做或漏做
2. 临时调试代码是否清理
3. 跑编译/检查，有没有报错

### 沟通

- 首次工具调用前用**一句话**说明要做什么
- 发现意外、改变方向、遇到阻塞 → **主动告知**
- 不做内部思考的流水账，只输出有用的进展

### 遇到错误

1. **先诊断根因**，不要用破坏性手段绕过（禁止 `--no-verify`、`reset --hard`）
2. 可逆地尝试修复 → 验证 → 继续
3. 修复失败 2 次以上 → 停下来，同步现状、列出选项
4. 复杂调试或多次修复失败 → 进入 `/systematic-debug` 四阶段流程，禁止猜测试错

### 遇到非当前任务的问题

| 发现的问题 | 处理方式 |
|-----------|---------|
| 和当前子任务相关，影响本次目标 | 记下来，做完当前再处理 |
| 与本次任务无关 | 记到一边，之后再说 |
| 影响面超出预期 | 停下来，说清楚，等确认 |

> **不改计划外的东西，除非问题已经确认。**

## 四、验证阶段

### 底线：验证必须跑

验证前务必回顾**验证铁律**：未经验证不得声明完成。最小验证要求见 `AGENTS.md`；性能与安全检查看 `quality-checklist.md`。

### 验证深度看风险

| 改动类型 | 验证方式 |
|---------|---------|
| 改变量名、类型标注 | 编译通过就行 |
| 改一个逻辑分支 | 想所有可能输入，跑一下 |
| 改核心逻辑 / 算法 | 边界、异常都想过一遍 |
| 改公开接口 | 检查所有调用方 |
| 改多个关联文件 | 确认整体行为一致 |
| UI / 前端改动 | 浏览器实际用一次，覆盖主路径 + 关键边界；无法实机验证**明确告知**用户 |

### 每个子任务完成后

- 对照完成条件逐条核对
- **完全满足**才标 `completed`；任何残缺保持 `in_progress` 并新建后续 todo

### 验证失败处理

1. 确认失败是否由本次改动引入，对照 diff 和日志
2. 能快速修复 → 修复后重跑验证
3. 不能快速修复 → 停止继续加改动，不靠猜测硬修
4. 失败信息有价值 → 写入 `TASK_MEMORY.md`
5. 如实说明失败项、已尝试动作和阻塞点；不说"应该可以"

### 整体完成前检查

- [ ] 所有用户显式要求已交付
- [ ] 必要验证已执行并记录结果
- [ ] 符合项目风格与规则
- [ ] 无遗留调试代码、console.log、TODO 占位
- [ ] 无多余改动（不相关文件未动）
- [ ] 对用户可见的变化已在回复中说明

## 五、收尾阶段

### 回复格式：说清三件事

1. **做了什么** — 解决了什么问题，不罗列每个文件改了哪几行
2. **有没有遗留** — 问题没处理 / 需要用户确认的，说清楚
3. **需要对方做什么** — 验证什么、怎么验证

引用文件用 `[filename](path)` 或 `[filename:line](path#Lline)`。

### 未完成事项

- 任何未完成 / 未验证 / 跳过的部分，**必须在回复中显式说明**
- 不使用"应该可以工作"的含糊话术；未测试就说未测试

### 记忆与持久化

- 用户偏好、项目背景、反馈意见 → 按类型写入 `AI_MEMORY.md` / `TASK_MEMORY.md`
- 临时状态、当前会话上下文 → 不入 memory
- 用户要求"永久记住 / 以后都这样" → 必须落盘到规范或 hooks
- 收尾前检查是否命中 `AI_MEMORY.md` 写入触发条件

---

## 六、元规则

1. **范围纪律**：用户授权 = 本次可执行范围，不扩张
2. **诚实优先**：不确定、未验证、未测试一律如实说明
3. **最小改动**：能改 1 行不改 10 行
4. **可逆偏好**：多个方案同效时选更可逆的
5. **用户节奏**：探索性问题 → 短答 + 推荐 + 主要权衡
6. **规范自举**：本规范可被用户要求修改；修改后立即按新版执行

---

## 七、快速自检口诀

> **理解 → 拆解 → 依赖 → 执行 → 验证 → 汇报**

每步问自己：

- 我真的明白用户要什么了吗？
- 验证证据是新的还是"上次跑的"？
- 这一步最小且可验证吗？
- 有没有更可逆的做法？
- 我说的和我做的一致吗？
