# 质量检查清单（Quality Checklist）

> 本文件定义**性能、安全与实现质量**约束。
> 类型检查、lint、测试、交付前验证看 `AGENTS.md` 与 `workflow.md`。

---

## 一、性能

### 加载性能

- 路由懒加载
- 大组件异步加载
- 图片懒加载
- 构建按需分包

### 运行时性能

- 静态内容使用缓存机制
- 大数据列表使用虚拟滚动
- 大对象使用浅响应/非响应式标记
- 高频事件（scroll/resize/input）加防抖/节流
- 动画使用 `requestAnimationFrame`

### 内存管理

- 组件卸载时清理：定时器、事件监听、实例、缓存
- 大对象避免深层代理

### 网络优化

- API 请求使用缓存策略
- 图片使用现代压缩格式（WebP/AVIF）
- 启用 gzip/brotli 压缩

### 性能指标（Core Web Vitals）

| 指标 | 目标 |
|------|------|
| FCP | < 1.8s |
| LCP | < 2.5s |
| TTI | < 3.8s |

**验证方式**：

- 开发联调：优先使用 Performance 面板做本地基线测量
- 提测前：至少选择一个核心页面执行一次性能测量
- 涉及真实网络、图片加载或首屏渲染的场景：优先在接近真机的设备和网络条件下验证
- 若本次改动不具备可测条件，必须在交付说明中明确标注"性能指标未实测"

## 二、安全

### 输入验证

- 所有表单输入使用 schema 验证
- 前端验证不可信，后端必须二次验证

### XSS 防护

- ❌ 禁止直接使用未经处理的 HTML 注入
- ❌ 用户输入不要直接插入 DOM

### 敏感数据处理

- Token 存储：httpOnly cookie 优先于 localStorage
- ❌ 禁止在前端硬编码密钥
- 敏感日志脱敏处理

### 文件上传

- 限制文件类型和大小（客户端 + 服务端双重校验）
- 服务端检查 MIME 类型，重命名上传文件

### 依赖安全

- 定期运行安全审计，及时修复有漏洞的依赖

## 三、代码组织

Reviewer 审查和 Developer 编码时均需关注：

- **函数拆分**：单函数 ≤50 行，超过应拆分；每个函数只做一件事
- **组件拆分**：单文件 ≤800 行，超过应提取子组件或工具模块
- **拆分合理性**：拆分粒度适中，避免过度拆分（每个文件仅 10-20 行且无独立复用价值）或拆分不足（一个文件包含多个独立职责）
- **职责单一**：每个模块/组件/函数的职责边界清晰，不混杂数据获取、业务逻辑和 UI 渲染
- **复用提取**：出现 3 次及以上的相同逻辑应提取为公共函数或组件；出现 2 次但业务语义完全一致、后续预计共同演进时也应考虑提取
- **封装克制**：只为真实复用、复杂度隔离或职责边界封装；禁止为单次使用、过早抽象或规避代码行数而制造无意义组件 / 函数

### Reviewer 实现质量门禁

功能跑通只是最低要求，不等于代码审查通过。Reviewer 必须审查“实现方式是否值得合入”，不能只查 bug。

Reviewer 必须基于 diff、相关文件和已确认方案判断：

- **方案一致性**：实现是否偏离已确认方案、接口契约、数据流或任务边界
- **既有模式一致性**：是否复用项目已有组件、工具函数、状态管理、错误处理和命名风格
- **可维护性**：是否存在临时拼接、重复逻辑、长函数、长组件、过度嵌套、隐式副作用、隐藏全局状态
- **抽象与封装质量**：是否遗漏应提取的通用组件 / 函数 / composable / 工具方法；是否存在过度封装、过早抽象、仅转发参数的薄封装、无复用价值的小文件堆叠
- **可读性**：命名是否表达意图，控制流是否清晰，复杂逻辑是否有合理拆分
- **类型与边界**：是否用 `any`、类型断言、空值绕过、吞异常等方式掩盖问题
- **最小改动**：是否引入无关重构、无关依赖、过宽影响面或难以回滚的改动
- **可测试性**：核心逻辑是否可独立验证，边界条件是否能被测试覆盖

判定规则：

| 严重级别 | 判定标准 | 处理 |
|----------|----------|------|
| CRITICAL | 破坏功能、数据安全、核心契约或引入不可接受风险 | 阻断 |
| HIGH | 功能可用但实现粗糙，明显破坏可维护性、项目模式、抽象边界或后续演进 | 阻断 |
| MEDIUM | 有局部质量问题但不阻断当前合入，可明确后续修复 | 记录并建议修复 |
| LOW | 命名、注释、细节风格等轻微问题 | 建议 |

若代码“功能实现了但写得差”，Reviewer 必须至少给出 HIGH 或 MEDIUM 问题，不得以“功能可用”直接通过。重复逻辑未提取、应该封装通用组件 / 函数却散落实现，或过度封装导致理解成本上升，都属于实现质量问题。HIGH 问题必须路由 developer 修复后复审。

Reviewer 输出必须包含：

```md
## Reviewer 结论

可合入：是 / 否
总体评价：[不仅描述功能是否正确，还要评价实现质量]

### 阻断问题
- [CRITICAL/HIGH] 文件:行 — 问题、原因、建议修复方向

### 非阻断建议
- [MEDIUM/LOW] 文件:行 — 问题、原因、建议

### 已检查但未发现问题
- [方案一致性 / 既有模式 / 可维护性 / 抽象与封装质量 / 类型边界 / 测试性]

### 剩余风险
- [没有则写“无”]
```

## 四、UI 审查

涉及页面 UI 改动时，Reviewer 必须借助 Playwright 进行可视化审查：

- 使用 Playwright 打开改动页面，截取关键状态截图（正常、空态、错误态、加载态）
- 对照设计稿或现有页面，检查布局、间距、颜色、字体是否一致
- 检查 hover、focus、active 等交互状态是否完整
- 检查响应式：至少验证 375px、768px、1024px 三个断点
- 检查暗色/亮色主题（如项目支持）
- 发现视觉差异或交互缺陷时，截图标注后附在审查意见中

> 纯逻辑改动、API 改动、后端代码可跳过 UI 审查。

## 五、重复问题修复要求

若修复后复审仍存在同类问题，Reviewer 不得只描述问题，必须给出**具体修复方案**：

- **页面样式问题** → 给出具体的 CSS 属性值或组件结构调整（不能只说"间距不对"）
- **功能逻辑问题** → 给出代码修改建议或伪代码（不能只说"逻辑有问题"）
- **拆分/架构问题** → 给出具体的重构方向和文件拆分建议

> 第 1 轮审查可以只指出问题；第 2 轮同类问题必须附带修复方案。

## 六、通用安全检查清单

提交前确认：

- [ ] 无硬编码密钥（API 密钥、密码、令牌）
- [ ] 所有用户输入已验证
- [ ] SQL 注入防护（参数化查询）
- [ ] XSS 防护
- [ ] CSRF 保护已启用
- [ ] 认证/授权已验证
- [ ] 错误消息不泄露敏感数据
