# 持续学习与 Skill 进化

> 目的：从重复任务和反复出现的问题中提取模式，自动进化为可复用的 Skill 或长期规则。
> 不是"记录流水账"，而是"发现模式 → 验证稳定性 → 固化规则"。

---

## 触发条件

满足以下任一条件时，触发 Skill 进化流程：

| 信号 | 阈值 | 说明 |
|------|------|------|
| 同类 bug 反复出现 | ≥ 2 次修复同一类问题 | 表明存在系统性缺陷或规则缺失 |
| 同类任务重复执行 | ≥ 3 次执行相同类型的任务 | 表明可提取为标准化 Skill |
| 反复踩同一坑位 | 同一类问题在不同任务中出现 | 应写入 AI_MEMORY.md 长期规则 |
| 规范被频繁绕过 | 同一规范条目多次被 agent 忽略 | 规范可能不合理或缺少 enforcement |
| 新工具/新框架引入 | 项目技术栈发生变化 | 需新增对应编码规范和 Skill |

---

## 观察记录（信号收集层）

进化流程需要一个前置步骤：**Agent 不能在触发条件命中时才去回忆，必须在每次疑似模式出现时立即记录。** 观察记录就是信号收集层——攒够阈值后自动进入 Phase 1。

### 记录位置

观察记录写入 `AI_MEMORY.md` 的「观察中模式」章节，不另建文件。一个模式一行，按以下格式维护：

```markdown
## 观察中模式（达到阈值后进入持续学习进化流程）

- **[N/阈值] [信号类型]：[一句话描述]**
  - 最近出现：[日期]（[任务简述]）
  - 特征：[关键共性（可选）]
```

| 字段 | 说明 |
|------|------|
| `N` | 当前观察到次数，每次命中 +1 |
| `阈值` | 来自触发条件表（如 bug 类 = 2，任务类 = 3） |
| `信号类型` | 同类 bug / 同类任务 / 同一坑位 / 规范绕过 / 技术栈变更 |

### 检查点（何时记录 / 更新计数）

每个任务收尾时必须做一次「模式回顾」，检查当前任务是否命中了已有观察项，或者产生了新的观察项。具体来说有两个动作：

**动作 1：更新已有观察项。** 遍历 `AI_MEMORY.md`「观察中模式」中的所有条目，判断本次任务是否属于同一种模式。如果是 → N+1，更新"最近出现"字段。如果 N 已经达到阈值 → 在当前会话或下个会话立即进入进化流程 Phase 1。

**动作 2：新增观察项。** 本次任务中是否出现了以下任一信号，但「观察中模式」中没有对应条目：

- 修了一个 bug，和前不久修过的很像 → 新增 `[1/2] [同类 bug]：xxx`
- 完成了一个任务，方法和之前 2 次高度相似 → 新增 `[1/3] [同类任务]：xxx`
- 踩了一个 `AI_MEMORY.md`「长期有效规则」里已记录的坑 → 新增 `[1/2] [同一坑位]：xxx`
- 某个规范条目本次任务中被绕过了 → 新增 `[1/3] [规范绕过]：xxx`

**检查点的具体位置**：

| 触发场景 | 检查点 |
|---------|--------|
| 每个任务收尾 | `session-end.md` 收尾清单中的「重复模式回顾」 |
| Phase 5 交付 | Orchestrator 在交付前确认 |
| 长对话中途（> 20 轮） | Orchestrator 主动做一次信息同步时顺便检查 |

### 达到阈值 → 进入进化流程

当某个观察项的 `N ≥ 阈值` 时：

1. 该条目从「观察中模式」移动到「持续学习进化流程」的通知区
2. Agent 在当前会话或下次会话启动时，按 Phase 1→4 执行进化
3. 如果连续 3 次未再出现，N 不自动降级，保留观察（人工判断是否删）

### 条目清理规则

- **过期**：连续 6 个任务未再出现 → 标注 `（待清理）`，下次维护时删除
- **升级**：达到阈值 → 移入进化流程，原观察条目保留到固化完成后删除
- **否定**：用户明确判定这不是模式 → 直接删除

### 模式回顾的输出格式

每次收尾时，Agent 输出：

```text
【模式回顾】
已有观察项：N 个
本次命中： [条目] → N：X → X+1（[是否达阈值]）
本次新增： [条目]
下次行动： [无 / 观察项 xxx 已达阈值，建议下次会话进入持续学习进化流程]
```

## 进化流程

> 以下四阶段仅在观察项达到阈值后执行。

### Phase 1：模式提取

1. **识别重复**：当前任务是否与历史任务存在相似性？
2. **抽象共性**：提取不变的部分（输入 → 处理 → 输出）
3. **记录差异**：标注哪些部分因场景不同而变化

### Phase 2：稳定性验证

模式必须经过以下验证才能固化：

- [ ] 至少 2 次独立场景中验证有效
- [ ] 不同 agent 执行时效果一致
- [ ] 未引入额外副作用或复杂度

### Phase 3：固化规则

根据模式类型，选择固化目标：

| 模式类型 | 固化目标 | 示例 |
|---------|---------|------|
| 任务执行流程 | 新增/更新 Skill | 提取"组件升级"通用流程为 skill |
| 编码约束 | 更新 coding-standards | 发现某类写法反复出问题 → 加入禁止项 |
| 架构坑位 | 更新 AI_MEMORY.md | 某第三方库时序问题 → 写入长期规则 |
| 工具链问题 | 更新 workflow.md 或新增脚本 | 某构建步骤反复失败 → 标准化处理流程 |
| 规范缺失/不合理 | 更新对应规范文件 | 某规则经常被绕过 → 检查规则是否合理 |

### Phase 4：通知与审查

- 向用户说明发现的新模式和拟固化的规则
- 用户确认后执行写入
- 重大 Skill 新增需走 `solution-brainstorm` 最小流程

## 写入规范

### Skill 新增/更新

新增 Skill 需包含：

- `SKILL.md`：完整流程定义
- 路由文件：`.claude/commands/{skill-name}.md`
- `skills.md` 注册：更新 Skill 清单

### AI_MEMORY.md 更新

按以下格式追加：

```markdown
- **[分类]** 规则结论。边界条件或失败后果
  - 来源：[任务描述或 issue 链接]
  - 验证次数：N 次
  - 相关文件：[路径]
```

### 规范文件更新

- 遵循 `spec-governance.md` 唯一来源原则
- 不重复已有规则，只补充缺口
- 交叉引用改为链接，不复制正文
- **重大新增/修改**走本文档的进化流程；**微调纠错**直接按 `spec-governance.md` 的修改检查清单即可

---

## 禁止行为

- ❌ 未经稳定性验证就固化（避免过早抽象）
- ❌ 将一次性问题写入长期规则
- ❌ 在多个文件重复记录同一模式
- ❌ 更新规范时不更新索引（AGENTS.md、skills.md 等）
- ❌ 不与用户确认就修改规范文件
