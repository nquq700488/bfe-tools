# Solution Brainstorm

> **硬门禁**：完整模式、高风险或需求不清的创造性工作，在呈现设计并获得用户批准之前，
> 不得调用实现 skill、编写代码或采取任何实现动作。
> **设计按复杂度缩放**：轻量任务一句话说清改什么即可继续；复杂任务走完整模式 A 或 B。

<HARD-GATE>
完整模式、高风险或需求不清时，在呈现设计并获用户批准前禁止进入实现。
</HARD-GATE>

## 「太简单不需要设计」是反模式

"简单"项目也可能被未经验证的假设浪费时间。设计可以很短——轻量改动一句话说清意图就是设计。只有完整模式、高风险或需求不清时，才把用户批准作为阻断点。

---

## 两种模式

| 模式 | 适用 | 特点 |
|------|------|------|
| **A：一对一设计细化**（默认） | 完整模式、高风险或需求不清任务 | 一次一问、2-3 方案、逐节确认、轻量高效 |
| **B：多模型/多角色头脑风暴** | 用户明确要求、全链路协作、跨模块高风险 | Round 1→2→3 收敛、多视角碰撞 |

从 A 开始。如果对话中自然需要多视角（需求模糊、跨模块、多个可行路径），可以升级到 B。

## 模式 A：一对一设计细化（默认）

### 步骤 1：探索上下文

先了解项目现状——check 相关文件、文档、最近提交。不做猜测。

### 步骤 2：评估范围

如果需求涉及多个独立子系统（如「建一个含聊天、文件存储、支付、分析的系统」），**立刻指出**——不要花时间细化一个需要先拆解的巨型项目。

先帮用户拆为独立子项目：哪些独立模块、什么关系、什么顺序做。然后对**第一个子项目**进入设计流程。

### 步骤 3：澄清问题（一次一问）

- **一次只问一个问题**，不等回复不跳下一步
- 优先出选择题（容易回答），必要时开放式
- 关注：目的、约束、成功标准、边界
- 不知道就问，不猜

### 步骤 4：探索方案

- 提 2-3 个不同方案，附权衡 + 推荐
- 推荐方案前置，说明理由
- YAGNI 原则：不必要的东西坚决删除

### 步骤 5：呈现设计（逐节确认）

按节呈现：架构 → 组件/模块 → 数据流 → 错误处理 → 测试策略。每节确认后再继续。每节篇幅按复杂度调整：简单的几句，复杂的最多 200-300 字。

设计原则：
- 每个单元一个清晰目的、定义良好的接口、可独立理解和测试
- 现有代码库中遵循既有模式；仅当影响本任务时才包含针对性改进
- 文件大小是信号：大文件往往意味着做了太多事

### 步骤 6：Spec 落盘

设计确认后，写入 `docs/specs/YYYY-MM-DD-<主题>-design.md`。（路径可被项目配置覆盖；如项目无 spec 目录，创建之或询问用户偏好路径。）这里只保存文件，不执行 git commit；提交必须由用户明确要求后走 `commit-helper`。

### 步骤 7：Spec 自检

落盘后立即自检：
1. **占位符扫描**：有没有 TBD、TODO、不完整段落、模糊需求？修复。
2. **内部一致性**：各节是否互相矛盾？架构是否匹配功能描述？
3. **范围检查**：是否聚焦到一份实现计划可以覆盖，还是需要再拆解？
4. **歧义检查**：有没有需求可以有两种解读？如有，选一种并明确写下来。

### 步骤 8：用户审核 Spec

> "Spec 已写入 `<路径>`。请审核，有任何想改的告诉我，之后再进入实现计划。"

等用户回复。有修改就改并重新自检。只有用户批准后才进入下一步。

### 步骤 9：过渡到实现

- 模式 A → 进入 `agent-workflow`（本地多 Agent 实现）
- 模式 B → 进入 `multi-agent-orchestrate`（CCB 多模型协作实现）
- 不得在本 skill 内直接编码。

## 模式 B：多模型/多角色头脑风暴

当用户明确要求多模型协作、全链路、「一起想方案」，或模式 A 中自然需要多视角时使用。

### 执行规则

1. 从 `.ccb/ccb.config` 读取可用 agent；CCB 不可用时降级到本地多角色。
2. 各 agent/角色从各自视角产出：

| 角色 | 视角 | 产出 |
|------|------|------|
| Planner | 方案设计 | 主方案、任务拆解、影响面、验收标准草案 |
| Inspiration | 弹性协作 | 替代方案、创新路径、易忽略方向 |
| Developer | 工程实现 | 技术可行性、实现风险、既有代码约束 |
| Reviewer | 质量审查 | 风险、反例、遗漏场景、约束冲突 |
| Tester | 测试视角 | 可测试性、关键边界、回归风险预判 |

3. 每轮 prompt 必须注入 `【身份设定】` + 按执行角色解析的 `【角色规范】`（见 `agent-registry.md`）。

### 轮次协议

| 轮次 | 目的 | 输入 | 输出 |
|------|------|------|------|
| Round 1 独立发散 | 防止从众，收集独立判断 | 需求、约束、相关文件 | 各角色独立方案/风险/替代路径/验收关注点 |
| Round 2 交叉挑战 | 看到彼此观点并碰撞 | Round 1 摘要矩阵 | 每个角色对其他观点的支持/反对/补充/待验证 |
| Round 3 收敛补强 | 消解关键分歧，形成可执行共识 | Round 2 分歧清单 | 修订建议、保留意见、共识确认 |

- 首轮全部并行提交；首轮完成后不得直接定稿，至少进入 Round 2
- 交叉挑战每轮最多 3 轮；连续一轮没有新增高价值分歧即停止
- 每轮每个 agent 必须明确回应其他观点：支持/反对/补充/待验证
- Orchestrator 主持记录，不直接消解重大分歧
- Round 3 后仍存在方向性分歧 → 输出分歧选项 + 代价 + 默认建议，交用户裁决
- 仅一个外部 agent 成功返回 → 告知"已降级为单外部视角 + 本地自检"

### 共识输出格式

```md
## 多模型头脑风暴共识

### 共识方案
- [最终建议的目标、范围、技术路径、任务拆解、验收标准]

### 采纳记录
| 来源 | 观点 | 裁决 | 理由 |

### 少数意见
- [未采纳但值得保留的观点、适用条件、风险]

### 未决问题
- [需用户确认或后续验证的问题；无则写"无"]

### 定稿动作
回传 planner 定稿 / 本地 Planner 定稿 / 需用户确认
```

共识后进入模式 A 的步骤 6-9（Spec 落盘 → 自检 → 用户审核 → 过渡实现）。

---

## 必守红线

- **硬门禁**：完整模式、高风险或需求不清的创造性工作，在呈现设计并获用户批准前不得进入实现。
- **轻量任务仍需说明意图**：一句话说清改什么即可，不强制用户确认和 Spec 落盘。
- 只有最外层 Orchestrator 可以启动本 skill；子 agent 不得递归启动。
- 模式 B 每次派发必须按 `agent-registry.md` 注入 `【角色规范】`。
- CCB 多模型时 agent/provider 只看 `.ccb/ccb.config`，角色语义只看 `agent-registry.md`。
- `inspiration` 代执行时必须按被替代角色注入规范和门禁。
- **完整模式 Spec 必须落盘**：未写入 design doc 的方案不算完成，不得进入实施。
- 不进入代码修改、review 放行或 test 放行。
- 模式 B 首轮观点 ≠ 完成头脑风暴，必须进入交叉挑战轮。
- 大项目先拆解子项目再做设计。
