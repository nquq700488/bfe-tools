# Test-Driven Development (TDD)

> **铁律**：如果你没看着测试失败，你就不知道它测试的是不是正确的东西。
> **违反规则字母 = 违反规则精神。**

---

## 铁律

```text
没有失败测试之前，不写任何生产代码
```

测试前写了本轮自己可安全丢弃的实现代码？删除。重新来。

**适用边界：**
- 只删除本轮自己新增且确认可重建的实现代码
- 不删除、不回退用户已有改动或工作区中来历不明的改动
- 无法安全区分时，先隔离当前改动或询问用户

**对可安全丢弃的本轮实现，没有例外：**
- 不能"留着当参考"
- 不能"边写测试边适配"
- 不能看
- 删除就是删除

从测试全新实现。

## 使用场景

**总是使用：**
- 新功能
- Bug 修复（先写复现测试）
- 重构
- 行为变更

**例外（需用户同意）：**
- 一次性原型
- 自动生成代码
- 纯配置文件

在想"就这一次跳过 TDD"？停。那是合理化借口。

## RED-GREEN-REFACTOR 循环

### RED — 写失败测试

写一个最小测试展示应该发生什么：

```typescript
// ✅ 好：清晰名称、测试真实行为、只测一件事
test('失败操作应重试 3 次', async () => {
  let attempts = 0;
  const operation = () => {
    attempts++;
    if (attempts < 3) throw new Error('失败');
    return '成功';
  };
  const result = await retryOperation(operation);
  expect(result).toBe('成功');
  expect(attempts).toBe(3);
});
```

```typescript
// ❌ 坏：模糊名称、测 mock 不测代码
test('重试可用', async () => {
  const mock = jest.fn()
    .mockRejectedValueOnce(new Error())
    .mockResolvedValueOnce('成功');
  await retryOperation(mock);
  expect(mock).toHaveBeenCalledTimes(3);
});
```

要求：一个行为、清晰名称、真实代码（尽量不用 mock）。

### 验证 RED — 看着它失败

**强制步骤，永不可跳过。**

```bash
# 按项目实际测试命令执行（示例）
npm test -- path/to/test.test.ts
```

确认：
- 测试失败（不是报错）
- 失败消息符合预期
- 失败原因：功能缺失（不是 typo）

**测试通过了？** 你在测已有行为。修正测试。
**测试报错了？** 修错误，重跑到失败正确为止。

### GREEN — 最小代码

写最简单的代码让测试通过。不添加功能、不重构其他代码、不"改进"超出测试范围的内容。

### 验证 GREEN — 看着它通过

**强制步骤。** 确认：测试通过、其他测试仍然通过、输出干净无误。

### REFACTOR — 清理

只在通过后：消除重复、改进命名、提取辅助函数。保持绿色。不添加行为。

### 重复

下一个失败测试 → 下一个功能。

## 为什么顺序重要

### "我可以写完之后再补测试验证它没问题"

事后写的测试立即通过。立即通过什么都证明不了——可能测的是错误的东西、可能测的实现细节、可能漏了你忘记的边界情况。你从来没看到它捉住 bug。

### "我已经手动测了所有边界情况了"

手动测试是临时性的。没有记录、不能重跑、压力下容易漏。"我当时试了能工作" ≠ 全面。

### "丢掉本轮实现太浪费了"

沉没成本谬误。对本轮自己新增且可安全丢弃的实现，当前选择是：隔离/丢掉后用 TDD 重写（高信心），或保留并事后加测试（低信心，可能有 bug）。如果改动包含用户已有内容或来历不明内容，先隔离或询问，不能直接删除。

### "TDD 是教条，务实意味着灵活"

TDD **就是**务实——提交前找 bug（比事后调试快）、防止回归（测试立即捉住破坏）、文档化行为（测试展示如何使用代码）、启用重构（自由修改，测试捉住破坏）。

## 常见借口

| 借口 | 现实 |
|------|------|
| "太简单不需要测试" | 简单代码也会崩。测试 30 秒。写。 |
| "我之后再测试" | 立即通过的测试什么都没证明。 |
| "事后测试能达到同样效果" | 事后测试回答"代码做什么"，测试先写回答"代码应该做什么"。 |
| "已经手动测过了" | 临时 ≠ 系统。无记录、不能重跑。 |
| "删除 X 小时太浪费了" | 沉没成本谬误。对本轮自己可安全丢弃的未验证实现，保留就是技术债。 |
| "留着当参考，测试先写" | 你会适配它。这是事后测试。可安全丢弃时就隔离或删除。 |
| "需要先探索" | 可以。探索代码完成后隔离或丢弃，用 TDD 从头实现。 |
| "TDD 会拖慢我" | TDD 比调试快。务实 = 测试先写。 |
| "测试不好写 = 设计不清晰" | 听测试的。不好测试 = 不好使用。 |

## 红牌 — 停下，重新来过

以下念头出现时必须停下来：

- 测试前写了代码
- 实现后补的测试
- 测试立即通过
- 解释不了为什么测试失败
- "稍后"加的测试
- "就这一次"的合理化
- "我已经手动测试过了"
- "事后测试能达到同样目的"
- "这是精神而不是仪式问题"
- "留着当参考"或"适配已有代码"
- "已经花了 X 小时，删除太浪费"
- "TDD 是教条，我这是务实"
- "这次不一样，因为..."

**所有以上情况意味着：停止继续实现。对本轮自己可安全丢弃的代码，隔离或删除后用 TDD 重新来过；涉及用户已有改动时先询问。**

## Bug 修复示例

**Bug**：空 email 被接受

**RED**
```typescript
test('拒绝空 email', async () => {
  const result = await submitForm({ email: '' });
  expect(result.error).toBe('Email 不能为空');
});
```

**验证 RED**
```bash
$ npm test
FAIL: expected 'Email 不能为空', got undefined
```

**GREEN**
```typescript
function submitForm(data: FormData) {
  if (!data.email?.trim()) {
    return { error: 'Email 不能为空' };
  }
  // ...
}
```

**验证 GREEN**
```bash
$ npm test
PASS
```

**REFACTOR** — 如果需要，提取多字段验证。

## 完成前检查清单

- [ ] 每个新函数/方法都有测试
- [ ] 每个测试在实现前都看着它失败了
- [ ] 每个测试失败的原因正确（功能缺失，不是 typo）
- [ ] 为每个测试写了最小代码
- [ ] 所有测试通过
- [ ] 输出干净（无错误、无警告）
- [ ] 测试使用真实代码（mock 仅限必要时）
- [ ] 边界情况和错误已覆盖

任何项没打勾 = 你跳过了 TDD。重新来过。

## 好测试

| 质量 | 好 | 坏 |
|------|----|----|
| **最小** | 一件事。名字里有"和"？拆开。 | `test('验证 email 和 domain 和空格')` |
| **清晰** | 名字描述行为 | `test('test1')` |
| **展示意图** | 展示期望的 API | 掩盖代码应该做什么 |

---

## 卡住时

| 问题 | 解决方案 |
|------|---------|
| 不知道怎么测试 | 写你想要的 API。先写断言。问用户。 |
| 测试太复杂 | 设计太复杂。简化接口。 |
| 所有东西都要 mock | 代码太耦合。用依赖注入。 |
| 测试准备代码太长 | 提取辅助函数。仍然复杂？简化设计。 |

---

## 必守红线

- **铁律**：生产代码 → 测试存在且先失败 → 否则不是 TDD。
- **没有例外可以不经用户同意**。
- 事后测试 ≠ TDD。有覆盖率，丢了证明测试有效的证据。
- 调试集成：发现 bug？先写复现测试。测试证明修复有效且防回归。
- 30 分钟事后测试 ≠ TDD。
