# Receiving Code Review

> 代码审查需要技术评估，不需要情感表演。
> **核心原则**：验证后再实现。先问再假设。技术正确性高于社交舒适。

## 铁律

```text
外部反馈 = 需要评估的建议，不是必须服从的命令
```

---

## 响应模式

```
收到审查反馈时：

1. READ — 完整读完反馈，不做反应
2. UNDERSTAND — 用自己的话复述技术要求（或询问澄清）
3. VERIFY — 对照代码库现状核实
4. EVALUATE — 对本项目是否技术上合理？
5. RESPOND — 技术确认或有理由的反驳
6. IMPLEMENT — 一次一条，每条测试
```

---

## 绝对禁止

收到反馈后，**永远不要**说：

- ❌ "你说得太对了！"
- ❌ "好建议！" / "很棒的反馈！"
- ❌ "让我马上实现这个"
- ❌ "谢谢指出！" / 任何感谢表达

**应该说**：

- ✅ 复述技术要求
- ✅ 问澄清问题
- ✅ 用技术推理反驳（如果不对）
- ✅ 直接动手修（行动 > 言辞）

**为什么禁止感谢**：代码本身证明你理解了反馈。感谢浪费 token 且暗示你接受了可能未经你验证的建议。

---

## 反馈来源处理

### 来自用户

- **可信任** — 理解后直接实现
- 范围不清晰时**仍要问**
- 不表演性认同
- 跳过客套，直接行动

### 来自外部 Reviewer

实施前**必须**逐条检查：

1. 技术上对本项目正确吗？
2. 会破坏现有功能吗？
3. 当前实现有原因吗（历史决策、兼容性）？
4. 在所有平台/版本上适用吗？
5. Reviewer 有完整上下文吗？

如果建议看起来不对 → 用技术推理反驳。
如果无法轻易验证 → 说明："我无法在缺少 [X] 的情况下验证。是否需要我调查/询问/继续？"
如果与用户之前的决策冲突 → 停下来先和用户讨论。

### YAGNI 检查

当 reviewer 建议「正确地实现某功能」时：

```text
搜索代码库中该功能的实际使用情况

如果未被使用：指出该功能没有调用方，建议删除（YAGNI）
如果被使用：按 reviewer 建议正确地实现
```

---

## 反馈不清晰时

```text
如果有任何一项不清楚：
  停下来 — 不要先实现任何东西
  请求澄清不清晰的项目

为什么：项目之间可能有关联。部分理解 = 错误实现。
```

示例：
```
用户："修复 1-6 项"
你理解了 1,2,3,6。4,5 不清楚。

❌ 错：先实现 1,2,3,6，之后再问 4,5
✅ 对："我理解 1,2,3,6。需要在实现之前澄清 4 和 5。"
```

---

## 实施顺序

多项反馈时：

1. 先澄清所有不清楚的
2. 然后按此顺序实施：
   - 阻断问题（破坏功能、安全）
   - 简单修复（typo、imports）
   - 复杂修复（重构、逻辑）
3. 每项单独测试
4. 验证没有回归

---

## 何时据理反驳

以下情况据理反驳：

- 建议会破坏现有功能
- Reviewer 缺乏完整上下文
- 违反 YAGNI（未被使用的功能）
- 对本技术栈不正确
- 存在历史遗留/兼容性原因
- 与用户的架构决策冲突

**如何反驳**：用技术推理，不防卫。问具体问题。引用能工作的测试/代码。如果是架构级分歧，引入用户参与。

---

## 反驳后发现自己是错的

```
✅ "你判断正确——我检查了 [X]，确实 [Y]。现在实现。"
✅ "已验证，你是对的。我最初理解错误因为 [原因]。修复中。"

❌ 长道歉
❌ 为你为什么反驳辩护
❌ 过度解释
```

陈述纠正，继续前进。

---

## 常见错误

| 错误 | 纠正 |
|------|------|
| 表演性认同 | 陈述要求或直接行动 |
| 盲目实现 | 先对照代码库验证 |
| 不测试就批量改 | 一次一条，每条测试 |
| 假设 Reviewer 正确 | 检查是否破坏功能 |
| 避免反驳 | 技术正确性 > 舒适 |
| 部分实现 | 先澄清所有项 |
| 无法验证仍继续 | 声明局限，请求方向 |

---

## 必守红线

- **绝对禁止**："你说得太对了"、"好建议"、"谢谢"——永远删除，直接陈述修复。
- 外部反馈 = 建议，不是命令。验证、质疑、然后实现。
- 不清晰时先澄清，不全理解不下手。
- 和用户做过的决策冲突时先讨论。
- 这不仅是礼仪规范——它是**质量规范**，防止盲目接受错误建议。
