# 系统化调试（Systematic Debugging）

> 本 skill 强制在修复前完成根因调查，禁止随机试错。
> 本地执行方法论见 `workflow.md`，测试编写见 `testing-guidelines.md`。

## 铁律

```text
未完成根因调查，不得进行任何修复
```

如果还没完成 Phase 1（根因调查），你不能提出修复方案。随机试错浪费精力和产生新 bug。针对症状的热修复是失败的调试。

## 必读规范

1. `AGENTS.md`
2. `.agents/workflow.md`
3. `.agents/testing-guidelines.md`（涉及编写回归测试时）

## 使用边界

使用本 skill：

- 测试失败、生产 bug、异常行为
- 性能问题、构建失败、集成问题
- **尤其**：时间压力下（紧急情况最容易让人想跳过调查直接试）
- **尤其**："快速看一下就行"的冲动出现时
- **尤其**：已经试了多次修复还没解决

不使用本 skill：

- 类型错误、lint 报错（直接修复即可）
- 明确的单行逻辑错误（原因一目了然）
- 用户已给出明确修复指令且原因已确认

## 四阶段流程

必须按顺序完成每个阶段，不得跳过。

### Phase 1：根因调查

**在任何修复尝试之前：**

1. **仔细读错误信息**
   - 不要跳过任何错误或警告
   - 完整读堆栈追踪，注意行号、文件路径、错误码
   - 错误信息本身经常就包含答案

2. **稳定复现**
   - 能可靠触发吗？每次都能吗？
   - 精确的复现步骤是什么？
   - 不能稳定复现 → 收集更多数据，不靠猜

3. **检查最近的变更**
   - `git diff`、最近提交
   - 新依赖、配置变更
   - 环境差异

4. **多组件系统中收集证据**
   - 对每个组件边界：记录进入和离开的数据
   - 验证环境/配置传递
   - 运行一次收集证据 → 分析确定哪个组件出错 → 再深入调查

5. **回溯数据流（根因追溯）**
   - 错误值从哪里来？
   - 谁用错误值调用了这里？
   - 继续向上追溯直到找到源头
   - **在源头修复，不在症状处修复**

   示例：
   ```
   症状：git init 在 source code 目录创建了 .git
   追溯：git init 的 cwd 为空字符串 → WorktreeManager 收到空 projectDir
        → Session.create() 传了空字符串 → 测试在 beforeEach 前访问了 context.tempDir
        → setupCoreTest() 返回 { tempDir: '' }
   根因：顶层变量初始化时访问了空值
   修复：tempDir 改为 getter，在 beforeEach 前访问时抛错
   ```

### Phase 2：模式分析

**先找模式，再修复：**

1. **找能工作的例子**
   - 在同一代码库中找类似但正常工作代码
   - 什么类似的代码能正常工作？

2. **和参考实现对比**
   - 完整读完参考实现，不要跳读
   - 理解整个模式，再应用

3. **找出差异**
   - 工作和不工作的之间有什么不同？
   - 列出每一个差异，不管多小
   - 不要假设"那个不可能有关系"

4. **理解依赖**
   - 还需要什么其他组件？
   - 什么设置、配置、环境？
   - 做了什么假设？

### Phase 3：假设与验证

**科学方法：**

1. **提出单个假设**：清楚陈述"我认为 X 是根因，因为 Y"
2. **最小化测试**：做最小改动验证假设，一次只改一个变量
3. **验证后再继续**：有效 → Phase 4；无效 → 提新假设
4. **不知道就说不知道**：不假装知道，寻求帮助，继续研究

### Phase 4：实施修复

**修复根因，不是症状：**

1. **先写失败测试**——用 `test-driven-development` skill 或者直接写回归测试
2. **实施单个修复**——解决已识别的根因，一次一个改动，不顺手重构
3. **验证修复**——测试通过了吗？其他测试未受影响吗？问题确实解决了？
4. **修复失败时**：
   - 停下来，数一数试了多少次
   - < 3 次：回到 Phase 1 重新分析
   - **≥ 3 次：停下来质疑架构**——每个修复都在揭示新的共享状态/耦合/不同位置的问题？→ 这不是失败的假设，这是错误的架构。跟用户讨论前不再试。
5. **添加多层防御**：在根因修复后，在每一层加验证，让同类 bug 不可能再出现

## 常见借口与反驳

| 借口 | 现实 |
|------|------|
| "问题很简单，不需要流程" | 简单问题也有根因。流程对简单 bug 也快。 |
| "紧急情况，没时间走流程" | 系统调试比猜测试错快得多。 |
| "先试这个，再调查" | 第一次修复就定了基调。从一开始就做对。 |
| "我看出问题了，直接修" | 看出症状 ≠ 理解根因。 |
| "再试一次修复"（已 2 次以上） | 3 次以上就是架构问题。质疑模式，不再修复。 |
| "我手动验证就行，不用测试" | 没测试的修复不长久。测试先写才能证明。 |

## 红牌 — 停下，回到 Phase 1

以下念头出现时必须停下来：

- "先快速修一下，之后再调查"
- "试试改 X 看有没有用"
- "同时改几处，跑测试"
- "跳过测试，我手动验证"
- "应该是 X，直接修"
- "不完全理解但这可能有用"
- 提出修复方案但没有回溯数据流
- **"再试一次修复"（已经试了 2 次以上）**
- **每次修复都揭示新问题，位置不同**

**所有以上情况：停止。回到 Phase 1。**

---

## 3 次失败后的架构质疑

以下信号表明是架构问题，不是单个 bug：

- 每个修复都揭示新的共享状态/耦合/其他位置的问题
- 修复需要"大规模重构"
- 每个修复都在别处产生新症状

**停下来质疑基本面：**
- 这个模式从根本上可靠吗？
- 我们只是在"惯性坚持"吗？
- 是否应该重构架构，而不是继续修复症状？

**在尝试更多修复之前跟用户讨论。**

## 快速参考

| 阶段 | 关键动作 | 成功标准 |
|------|---------|---------|
| **Phase 1** 根因 | 读错误、复现、查变更、追溯数据流 | 理解 WHAT 和 WHY |
| **Phase 2** 模式 | 找能工作的例子、对比 | 找出差异 |
| **Phase 3** 假设 | 提理论、最小化测试 | 确认或提出新假设 |
| **Phase 4** 修复 | 写测试、修复、验证、加防御 | bug 解决，测试通过 |

## 补充技术

以下技术是本 skill 的内置摘要；如需要更完整的案例，可从上游 Superpowers 的 `systematic-debugging` skill 补充附件。

### 根因追溯（Root Cause Tracing）

- 错误出现在调用栈深处 → 向后追溯直到找到原始触发点 → 在源头修复
- 无法手动追溯时：在可疑操作前加 `console.error` 打印栈追踪和上下文
- **永远不在错误出现的地方直接修复，除非已经确认那确实是根因**

### 多层防御（Defense-in-Depth）

根因修复后在每一层加验证，让同类 bug 不可能再出现：

- 修复后问：如果再有人犯同样的错误，在哪些环节可以拦住？
- 在参数入口、中间层、最终执行处各加一道检查
- 每层验证失败都要有清晰的错误信息说明为什么失败和如何改正

### 条件等待（Condition-Based Waiting）

测试或异步代码中用条件轮询替代任意超时：

```typescript
// ❌ 任意超时 — 不可靠且慢
await new Promise(r => setTimeout(r, 1000));

// ✅ 条件等待 — 可靠且快
await waitFor(() => expect(result).toBe(expected), { timeout: 5000 });
```
