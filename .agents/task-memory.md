# 任务记忆规范

> `TASK_MEMORY.md` 记录**决策背景**（为什么这么做），是跨会话的经验积累。

---

## 文件结构

```markdown
# 任务记忆
> 描述...

---
## 最新任务 [状态]    ← 新任务插入到这里，时间倒序

---
## 旧任务 [已完成]
```

新任务插入到**第一个 `---` 之后**，禁止插入到标题前面。

---

## 什么值得记

**不是每个任务都写。** 主规则只有一条：

- **如果不记录，后续继续任务时需要重新分析上下文，就写。**

常见触发场景：

1. **跨模块影响**：修改了公共组件、工具函数、Store、Composables 等被多处依赖的代码
2. **架构/方案决策**：有明确的方案对比和取舍（为什么选 A 不选 B）
3. **复杂 Bug 修复**：根因非显而易见，或修复过程踩了坑，未来可能复发
4. **协议/契约变更**：影响接口格式、数据存储结构、交互协议
5. **长流程任务**：预计会跨会话继续，或中途需要多次恢复现场

**不需要记的**：机械性的多文件小改、单文件拼写或样式微调、纯配置加减（如 ESLint 规则）、无兼容性问题的依赖升级、文案改动。

## 记录格式

```markdown
## 任务标题 [状态]
目标：一句话描述
模型：单角色写模型名；多角色用 `/` 分隔（如 `方案 Opus 4.6 / 实施 Kimi Code CLI`）
决策：
- 方案：选择了什么 — 理由
- 排除：排除了什么 — 理由
问题/踩坑：
- [进行中/待验证] [已解决/待议] xxx
- [已完成] 关键词1、关键词2、关键词3
约束：
- 涉及文件：路径列表
- 其他限制（可选）
```

**只记三件事**：决策（选了什么）、风险（问题/踩坑）、约束（边界在哪）。验证检查项由 Reviewer 负责，不在此重复。

**字段命名约定**：

- `[进行中]` / `[待验证]`：使用 `问题：`，可写 `[已解决/待议]` 状态。
- `[已完成]`：使用 `踩坑：`，写关键词，不写长段过程。
- `涉及文件` 必写；无其他限制时可省略"其他限制"。

**默认不记**：

- 大段代码片段
- 逐步操作流水账
- 与当前判断无关的背景描述

**模型角色**：`方案`（架构决策）、`实施`（编码）、`审查`（代码审查）、`复核`（验证确认）。

---

## 状态与精简

| 状态 | 含义 | 记录要求 |
|------|------|---------|
| `[进行中]` | 编码中 | 保留目标、当前判断、关键决策、阻塞点、影响文件 |
| `[待验证]` | 编码完成，待复核 | 保留目标、待验证范围、关键风险、影响文件（不写执行流水） |
| `[已完成]` | 复核通过 | **精简摘要**：仅保留目标+模型+核心决策+踩坑关键词+文件列表，≤15 行 |
| `[已搁置]` | 暂停 | 保留恢复任务所需的最小上下文 |

流转：`[进行中]` → `[待验证]` → `[已完成]`，简单修复可跳过 `[待验证]`。

**排查期例外**：如果任务仍处于高频排查阶段，可以暂时保留更详细的定位信息；待问题收束后再精简。

**精简规则**：转 `[已完成]` 时，删除代码片段、before/after、详细步骤；保留目标（一句话）、模型、核心决策（各一句话）、踩坑关键词、文件列表。

## 示例

### [已完成]（精简摘要，≤15 行）

```markdown
## Mock 服务 404 修复 [已完成]
目标：修复开发环境 API 请求全部 404
模型：Kimi Code CLI
决策：
- 方案：`mockServerPlugin` 改为 `dir: 'src/mock'` + `include: '**/*.mock.ts'` — 插件 v2 默认 dir 不匹配
- 排除：移动 mock 文件到根目录 — 改动大且破坏结构
踩坑：v2 插件 include 路径相对于 dir 而非项目根
约束：涉及 `build/config/index.ts`，Mock URL 必须带项目路径前缀
```
